# Recommendation Candidate Pipeline

## Evidence To Candidate

A candidate should include:

- deterministic or stable candidate identity
- candidate type
- source evidence references
- evidence grade
- priority suggestion
- confidence suggestion
- risk flags
- seed risk
- client-facing draft when safe
- remeasurement signal
- draft-time rejection path

## Candidate Generation Must Not

- decide `auto_publish`
- convert provider errors into visibility gaps
- use README claims as client evidence
- use citation count as claim support
- create a client-facing candidate without traceable evidence
- hide missing fields
- infer strategy conclusions that belong to `georader-ai-search-auditor`
- override gate decisions from `recora-recommendation-quality-gate-auditor`

## Unsafe Inputs

Unsafe inputs should become rejection records or internal holds, not ordinary publishable candidates:

- missing measurement IDs for measurement claims
- unknown seed risk for discovery claims
- unknown source-to-claim status for citation-based candidates
- provider timeout or blocked state
- malformed or empty answer
- README/API/login/browser claims
- duplicated candidate generated by repeated job delivery

## Review Questions

- Is candidate identity stable across re-runs?
- Are duplicate candidates merged or superseded safely?
- Does the candidate reference raw evidence indirectly through evidence ledger IDs?
- Are draft-time rejections auditable?
- Is publication blocked until an approved gate review exists?
