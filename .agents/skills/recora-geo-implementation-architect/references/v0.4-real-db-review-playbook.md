# v0.4 Real DB Review Playbook

## CONFIRMED_FROM_FILES

- v0.3 implementation architect skill already requires distinguishing confirmed code facts from assumptions and preserving stack boundaries.
- v0.4 calibration has inspected a copied snapshot with migrations, candidate output, and generator code.

## USER_PROVIDED_CONTEXT

- v0.4 must inspect or explicitly mark unavailable migrations, schema, Supabase types, RLS, indexes, and persistence targets before proposing DB changes.

## NEEDS_VERIFICATION

- Any repo not matching the inspected snapshot must be audited again.
- Generated types, runtime DB state, policies, and app read/write paths must be rechecked before implementation.
- DB schema not fully inspected. Exact tables, columns, constraints, policies, and indexes remain NEEDS_VERIFICATION.

## RECOMMENDED_ARCHITECTURE

Use this review sequence:

1. Identify the repository root and copied snapshot boundary.
2. Inventory schema evidence: migrations, generated types, ORM schema, SQL snippets, seed data, candidate output, and generator code.
3. List missing schema evidence explicitly.
4. Map actual tables to RECORA concepts without inventing columns.
5. Separate raw measurement evidence, generated candidates, source-to-claim reviews, gate reviews, and published recommendations.
6. Check RLS and policy evidence before any tenant-safety claim.
7. Check idempotency and versioning before approving regeneration.
8. Check backfill and migration safety before proposing schema changes.
9. Produce review-mode file-impact maps and acceptance criteria only.
10. Refuse migration edits, DB commands, production ops, external APIs, crawler execution, browser automation, or quality-gate publication decisions unless a later task explicitly authorizes the exact scope.

## DO_NOT_ASSUME

- Do not use the v0.4 patch as proof that the production database is safe.
- Do not treat output JSON as schema.
- Do not collapse the strategy skill, implementation architect, and quality-gate auditor roles.
