# v0.3 Implementation Review Playbook

## CONFIRMED_FROM_FILES

- The implementation architect skill already requires code inspection before advice.
- The real candidate generator and exports now provide calibration fixtures.
- The current candidate JSON shape is rich but not aligned exactly to the stack contract.

## USER_PROVIDED_CONTEXT

- v0.3 should make the skill better at reviewing actual candidate generator code, schema gaps, provenance, parser/extraction risks, quality-gate handoff gaps, tests, and file-impact maps.

## NEEDS_VERIFICATION

- DB schema not inspected. Exact columns remain NEEDS_VERIFICATION.
- Application file layout and production module boundaries were not inspected for this patch.

## Review Procedure

1. Start with an artifact inventory.
2. Separate `CONFIRMED_FROM_FILES`, `USER_PROVIDED_CONTEXT`, `NEEDS_VERIFICATION`, `RECOMMENDED_ARCHITECTURE`, and `DO_NOT_ASSUME`.
3. Map actual output fields to the stack candidate contract.
4. Identify missing fields as:
   - absent
   - present under a different name
   - present under nested evidence
   - present but semantically different
5. Review source-to-claim handling before approving any citation candidate architecture.
6. Review seed contamination propagation before approving any discovery/absence candidate architecture.
7. Review provider and provider status separately from answer quality.
8. Review raw evidence and derived candidate boundaries.
9. Review quality-gate review persistence separately from generator scores and display fields.
10. Review idempotency and regeneration before suggesting persistence.
11. Produce a file-impact map without editing app code unless explicitly authorized later.

## File Impact Map Template

Use this template in future implementation reviews:

| Area | Likely file/module | Change type | Risk | Tests |
|---|---|---|---|---|
| Candidate schema | NEEDS_VERIFICATION until repo inspected | additive contract alias | backward compatibility | schema fixtures |
| Candidate generator | inspected generator or production equivalent | add fields/version/risk flags | output consumers | golden JSON |
| Source-to-claim | NEEDS_VERIFICATION until module inspected | new review object | citation safety | source fixtures |
| Quality gate persistence | NEEDS_VERIFICATION until schema inspected | separate persisted reviews | publication safety | history tests |
| Parser/extraction | NEEDS_VERIFICATION until parser inspected | span/version/seed risk | false positives | parser fixtures |
| Idempotency | NEEDS_VERIFICATION until jobs inspected | stable keys/generation run | duplicate candidates | retry tests |

## DO_NOT_ASSUME

- Do not name production modules that were not inspected.
- Do not propose exact migrations until schema and migration history are inspected.
- Do not implement from this patch package.
