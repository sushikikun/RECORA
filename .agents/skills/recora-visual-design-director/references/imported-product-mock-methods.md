# Imported Product Mock Methods

These methods adapt external dashboard, LP, and UI-review skill patterns into Recora-owned product mock direction. Do not copy external screenshots, HTML, CSS, icons, or layouts.

## Believable Layout Hierarchy

A mock should look like a product because its information architecture is believable:

1. topbar or sidebar
2. scope/filter row
3. KPI row
4. main table/ranking/list
5. log/source/insight panel
6. recommendation/status strip

If a mock lacks at least three of these layers, it usually reads as decoration.

## Role Of Each Layer

- Sidebar/topbar: anchors the surface as a product; keep nav short.
- Filter/scope row: explains sample status, query group, date, or report scope.
- KPI: summarizes state with rounded sample values.
- Table/ranking: proves the product can compare entities or queries.
- Log/source panel: shows evidence and source context.
- Insight panel: connects evidence to next action.

## Card Density Rule

- Large hero mock: 5-7 visible UI objects.
- Feature-card mini mock: 1 compact UI object.
- Sample report preview: 4-5 report objects.
- Final CTA ghost mock: silhouette only; no new claims.

## Fake Data Realism Vs Sample Clarity

- Use realistic shapes, not realistic client claims.
- Round numbers and label sample values.
- Keep company/provider names generic unless Product Truth confirms.
- Do not use customer logos, testimonials, outcomes, or precise benchmark values.

## Mini UI Not Decorative

Every mini UI must answer one question:

- Which queries are checked?
- Which source domains appear?
- Which competitor is visible?
- Which claim is supported?
- Which recommendation is awaiting review?

Decorative lines, random bars, and fake charts without a question should be removed.

## Dashboard Screenshot Illusion

Use the illusion of a dashboard without pretending it is a live connected account:

- browser or app frame
- clear title
- sample pill
- consistent rows
- readable labels
- subtle scroll/crop only when intentional

## Avoiding Random Cards

Random cards fail when they contain unrelated numbers, generic "AI score", identical card weights, no source/query context, or no next action. Replace them with a blueprint.

## Mobile Crop Strategy

- Hero Large Product Mock: KPI pair + 3-row ranking + one source panel.
- Feature Card Mini Mock: keep one UI object, reduce labels.
- Sample Report Preview: paper header + KPI + table excerpt + method note.
- Final CTA Ghost Mock: hide details; keep silhouette and CTA.

## Recora Mock Transformations

- Hero Large Product Mock: scope row, KPI, query/source table, insight panel.
- Feature Card Mini Mock: mini table, answer log, domain row, ranking bar, or heatmap.
- Sample Report Preview: report header, sample label, KPI, comparison table, source note, next action.
- Final CTA Ghost Mock: faint product silhouettes at low opacity, without data claims.
