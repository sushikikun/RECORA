# Anti-Patterns

Use this reference as a final check before answering.

## Scope Anti-Patterns

- Turning specialist analysis into GEORADER implementation.
- Editing app, LP, backend, CLI, database, production, payment, email, or deployment files.
- Mixing SNS consulting into GEORADER positioning.
- Treating generic SEO advice as sufficient for AI search visibility.
- Ignoring SEO foundations when the task asks for GEO/AI search strategy.
- Producing a polished report that lacks evidence labels, severity, or next actions.

## Evidence Anti-Patterns

- Calling GEORADER's framework an official GEO standard.
- Claiming a ranking factor, AI citation behavior, crawler behavior, or platform standard without official or research-backed support.
- Claiming runtime behavior without runtime evidence.
- Claiming browser rendering or SERP state was checked when it was not.
- Inventing AI answer results, citations, competitors, SERP results, scores, statistics, or sources.
- Treating a cited URL as supportive without checking whether it supports the claim.
- Hiding bad scores to make the report feel nicer.

## Trust Anti-Patterns

- Asking for `.env`, API keys, cookies, login sessions, credentials, or private tokens.
- Saying "submission received", "diagnosis complete", "report delivered", or "payment confirmed" without verified support.
- Letting sample or mock data look like the client's real result.
- Using client-facing language that overpromises ranking, AI recommendation, or guaranteed visibility.

## Output Anti-Patterns

- Putting summary before findings in a review.
- Omitting P0/P1/P2 when prioritization is requested or needed.
- Failing to separate internal-only notes from client-facing text.
- Giving vague advice such as "add more content" without tying it to intent, authority, citation readiness, or conversion.
- Ending with code-change instructions when the user asked for an audit, strategy, or report.
