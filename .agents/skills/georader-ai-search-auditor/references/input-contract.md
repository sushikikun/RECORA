# Input Contract

Use this reference before specialist work when inputs are incomplete.

## Minimum Inputs By Mode

| Mode | Minimum Useful Input |
|---|---|
| GEORADER strategy | Target area such as report, route, CLI, diagnosis logic, product roadmap, page, docs, or artifact |
| Client GEO/SEO/AI search strategy | Brand name plus domain or category |
| Technical discoverability review | Domain/page URL, crawl export, rendered HTML, screenshot, or code evidence |
| Search intent map | Brand/company, category, audience, region, and competitor list if known |
| Topical authority plan | Category, current content inventory, target audience, and conversion goal |
| Entity/brand authority review | Brand, domain, company info, author/operator proof, and third-party profiles if known |
| AI answer analysis | One or more answer texts, result rows, screenshots transcribed by the user, or exported result data |
| Competitor SERP/AI gap | Brand plus competitors, market/category, or supplied SERP/AI answer evidence |
| Report quality review | Report text, template, screenshots, JSON, or sample report |
| Trust-risk audit | UI/report/form text, route, screenshot description, or code evidence |

## Recommended Inputs

- Brand name and domain.
- Company, category, industry, region, target audience, use case, and price range.
- Competitors and indirect alternatives.
- Existing site pages, report samples, AI answer outputs, cited URLs, and SERP evidence.
- Search Console, analytics, crawl, or rank data when the user can safely provide exports.
- Whether output should be internal, client-facing, or both.

## Missing Data Handling

Do not block unless the target is completely unknown. When data is missing:

1. State assumptions clearly.
2. Use placeholders such as `{BRAND}`, `{DOMAIN}`, `{CATEGORY}`, `{REGION}`, and `{COMPETITOR_1}`.
3. Provide a usable draft and mark what should be verified later.
4. Avoid fabricated AI answer results, citations, scores, competitors, screenshots, runtime behavior, SERP state, ranking data, indexability state, or backend status.

## Verification Language

Use:

- `CONFIRMED_FACT` when evidence is present.
- `OFFICIAL_SOURCE` only for checked official or primary-source guidance.
- `RESEARCH_BACKED` only for named credible research.
- `INDUSTRY_PRACTICE` for common SEO/content/CRO practice.
- `GEORADER_ASSUMPTION` for GEORADER-specific strategy or framework.
- `NEEDS_VERIFICATION` when the claim needs current data, official source, runtime evidence, or user-provided proof.
